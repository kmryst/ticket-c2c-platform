# 0012. リフレッシュトークン（opaque + DB hash 保存 + rotate-on-use）で認証を強化する

## ステータス

Accepted

## 日付

2026-07-06

## 背景

ADR-0010 で導入したメール+パスワード認証は、アクセストークン（JWT HS256、1h）のみの構成である。production-readiness L-9 として次のギャップが記録されている。

- トークン失効（強制ログアウト・漏洩時の即時無効化）が原理的にできない。漏洩したトークンは最長 1h 有効なまま。
- 認証系エンドポイント（signup / login）にレート制限がなく、credential stuffing / パスワード総当たりを緩和できない。
- JWT シークレットのローテーション運用が未整備。Secrets Manager 上の値を更新して再デプロイすると、旧シークレットで署名された発行済みトークンが即座に全滅する（全ユーザー強制ログアウト）。

アクセストークンの寿命を単純に短縮すると UX が悪化する（15 分ごとに再ログイン）。寿命短縮とログイン状態の維持を両立するにはリフレッシュトークンが必要になった。

リフレッシュトークンの保存方式として、次の 3 案を比較した。

1. **JWT リフレッシュトークン（stateless）**: 失効リストがない限り即時失効できず、L-9 の核心（失効できない）を解決しない。
2. **opaque トークンを Valkey に保存**: 高速だが、Valkey はこのシステムで「落ちても fail-open で継続する非正本」と位置づけている（M-1/M-2）。失効情報を Valkey に置くと、Valkey 障害＝全ユーザーログアウト、または fail-open＝失効の保証喪失、の二択になる。
3. **opaque トークンの hash を PostgreSQL に保存**: 失効の正本が DB になり、確実に失効できる。refresh は 15 分に 1 回程度の低頻度アクセスなので DB 負荷は無視できる。

## 決定

リフレッシュトークンを導入し、次の設計とする。

- **opaque トークン + DB hash 保存**: リフレッシュトークンは 256bit のランダム値（base64url）とし、DB の `refresh_tokens` テーブルには SHA-256 hash のみ保存する。DB が漏洩しても元のトークンは復元できない。Valkey は使わない（案 3）。
- **rotate-on-use**: `POST /auth/refresh` が成功するたびに新しいアクセストークン + 新しいリフレッシュトークンを発行し、使用済みトークンは `revoked_at` / `replaced_by` で失効させる。
- **reuse detection**: 失効済み（使用済み・logout 済み・rotate 済み）トークンが再提示された場合は盗用の兆候とみなし、同一トークンファミリー（login/signup 時に採番する `family_id`）の全トークンを失効させる。正規ユーザー・攻撃者のどちらがトークンを持っていても、以降そのファミリーでは refresh できず再ログインが必要になる。単純な期限切れ（`expires_at` 超過）は盗用兆候ではないため、401 を返すだけでファミリー失効はしない。
- **rotation は DB トランザクション + 行ロックで原子化**: `/auth/refresh` は対象トークン行を `SELECT ... FOR UPDATE` でロックしてから状態を確認し、旧トークンの失効（`used_at` / `replaced_by_token_id`）と新トークンの INSERT を同一トランザクションで行う。同時 refresh の片方が「使用済み再提示」と誤判定されるレースをサーバ側でも排除する。
- **logout はリフレッシュトークンを失効させる**: 既存の `POST /auth/logout`（Cookie 削除のみ）を拡張し、提示されたリフレッシュトークンのファミリーを `revoked_reason = 'logout'` で失効させる。JWT アクセストークンは引き続き即時失効できない（最大 15 分残存）が、これは寿命短縮とセットで許容する。
- **アクセストークンは 15 分へ短縮**: 漏洩時の有効窓を 1h → 15 分に縮める。UX はフロントエンドの silent refresh（期限切れ時に自動で `/auth/refresh` を呼ぶ）で維持する。
- **リフレッシュトークンの寿命は 14 日**: rotate-on-use のため実質「最後の利用から 15 分間隔で延命」ではなく、ファミリー単位の絶対寿命は各トークンの `expires_at`（発行から 14 日）で制限する。
- **Cookie はアクセス用・リフレッシュ用で分離する**: `access_token` Cookie（Path=/、Max-Age=15 分）とは別に、リフレッシュトークンは `refresh_token` Cookie（Path=/api/auth、Max-Age=14 日）で発行する。Path を `/api/auth` に絞ることで、リフレッシュトークンが通常の API リクエストに同乗せず、送信先が refresh / logout 系エンドポイントに限定される。JSON body でも `refreshToken` を返し、非ブラウザクライアント（k6 / smoke script）は従来の `accessToken` 互換のまま利用できる。
- **CSRF 対策は SameSite=Lax + Path 限定を防御線とする**: `/auth/refresh` / `/auth/logout` は httpOnly Cookie で認証されるため CSRF 面を持つが、両 Cookie とも `SameSite=Lax` でありクロスサイトの POST には同乗しない。Origin/Host ヘッダ検証も検討したが、CloudFront 統合オリジン構成では ALB 到達時の Host が配信ドメインと一致する保証がなく（検証が誤って正規リクエストを弾くリスク）、また refresh の成果物（新トークン）は Cookie 経由でしか攻撃者に渡らないため、SameSite=Lax を前提とし Origin 検証は導入しない。SameSite 非対応の旧ブラウザサポートが要件になった場合は再検討する。
- **レート制限の送信元 IP は「信頼できる末尾ホップ数」を固定して決める**: `X-Forwarded-For` を無条件に信用せず、`RATE_LIMIT_TRUSTED_PROXY_HOPS`（インフラが末尾に追加する既知のプロキシ段数。dev/staging は CloudFront→ALB 構成で 1）で右から数えた位置の値をクライアント IP とする。ヘッダが取れない場合は TCP 接続元へフォールバックする。
- **認証系レート制限**: signup / login / refresh に IP 単位 + メールアドレス単位の固定ウィンドウカウンタ（Valkey Lua、M-2 で確立したパターンの再利用）を導入する。レート制限は「失効の正本」ではなく緩和策なので、Valkey 障害時は fail-open（制限なしで継続）でよい。
- **JWT シークレットの current/previous 二重検証**: Secrets Manager のシークレット値を `{"current": "...", "previous": "..."}` の JSON にし、検証は current → previous の順でフォールバックする。署名は常に current。これによりローテーション直後も発行済みトークンが即死しない。

## 根拠

- **失効の正本を DB に置く**: このリポジトリは「PostgreSQL が正本、Valkey は落ちてよい前段」という責務分離を一貫して採っている（ADR-0004、M-1/M-2）。セキュリティ判定（失効・reuse 検知）を fail-open にはできないため、正本 DB に置くのが方針に整合する。refresh の頻度はユーザーあたり 15 分に 1 回で、購入 API のような負荷特性を持たない。
- **opaque + hash 保存**: リフレッシュトークンは自己記述である必要がなく（サーバ側で引くだけ）、JWT にする利点がない。hash 保存により DB ダンプ漏洩時もトークンは無効化できる（bcrypt ではなく SHA-256 なのは、トークン自体が 256bit の一様ランダム値で総当たり不能なため。コスト付き hash は不要でレイテンシだけ増える）。
- **rotate-on-use + family**: OAuth 2.0 Security BCP（RFC 9700）の refresh token rotation の標準形。盗まれたトークンが 1 回使われた時点で正規側との再利用衝突が起き、ファミリー失効で被害を止められる。
- **15 分 / 14 日**: アクセストークン 15 分は「漏洩窓を 1/4 に縮めつつ、silent refresh の頻度が過剰にならない」折衷。リフレッシュトークン 14 日は「2 週間使わなければ再ログイン」という一般的なコンシューマ向け UX。
- **レート制限を Valkey に置く**: カウンタは短命（ウィンドウ 15 分程度）かつ失っても安全側でない影響が限定的（制限が一時的に外れるだけ）。既存の ElastiCache と Lua script 資産（M-2）を再利用でき、追加コストゼロ。
- **current/previous 二重検証**: 検証側だけ 2 本受けることで、「新シークレットへの切替 → アクセストークン最大寿命（15 分）の経過 → previous の破棄」という無停止ローテーション手順が成立する。コード変更は検証フォールバックのみで小さい。

## 反対材料・トレードオフ

- **DB 書き込みの増加**: login / refresh のたびに `refresh_tokens` へ INSERT/UPDATE が走る。使い捨て row が蓄積するため、期限切れ row の定期削除（cleanup）をいずれ整備する必要がある（当面はテーブルサイズが問題になる規模ではない）。
- **reuse detection の誤爆**: ネットワーク障害等で「サーバは rotate 済みだがクライアントは旧トークンしか持っていない」状態になると、正規ユーザーでもファミリー失効して再ログインになる。安全側に倒す設計として許容する。
- **レート制限の IP 単位はスプーフィング耐性に限界がある**: trusted-hops 方式でも、CloudFront を経由せず ALB を直接叩く経路では偽装 X-Forwarded-For を排除しきれない。IP 制限は「単純な単一送信元の総当たり」への緩和と割り切り、メールアドレス単位の制限（偽装不能）を主防衛線とする。厳密な送信元特定が必要になったら WAF レートベースルールへ移行する。（**2026-07-06 追記**: この「CloudFront を経由せず ALB を直接叩く経路」自体を ADR-0013 / Issue #190 で遮断した。ALB SG ingress を CloudFront origin-facing managed prefix list に限定したため、外部からの到達は必ず CloudFront を通り、trusted-hops による IP 判定の前提が成立するようになった。ここで述べた IP バイパスの既知の制約は解消済み。）
- **reuse detection の誤爆はクライアント側の抑制も前提**: 複数タブの同時 401 で refresh が並走すると、rotate-on-use の性質上どちらかが「使用済み再提示」になる。サーバ側の行ロックに加え、フロントエンドで refresh を single-flight 化（同時 401 でも `/auth/refresh` は 1 回だけ実行し他はその Promise を待つ）して発生源を抑える。
- **セッション管理画面はスコープ外**: 「ログイン中のデバイス一覧・個別失効」のような UI は作らない。`family_id` 設計により将来の拡張は可能。

## 再検討のトリガー

- refresh の QPS が DB のホットパスになったとき（Valkey へのキャッシュ層追加、または専用ストアを検討）。
- `refresh_tokens` テーブルの肥大が運用問題になったとき（TTL cleanup job / パーティショニングの導入）。
- 認証系への攻撃が IP 偽装を伴う規模になったとき（AWS WAF レートベースルールへの移行）。
- MFA・デバイス管理が要件になったとき（ADR-0010 の Cognito 移行トリガーと同じ損益分岐点の再評価）。
