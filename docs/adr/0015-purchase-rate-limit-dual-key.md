# 0015. 購入エンドポイントのレート制限を user_id 主体の dual-key 方式で導入する

## ステータス

Accepted

## 日付

2026-07-07

## 背景

購入エンドポイント（POST /events/:eventId/purchases）にはレート制限がなく、認証さえ通ればアプリ層では無制限にリクエストできた。人気イベントの発売時にはボットによる購入リクエストの物量攻撃が想定される（チケット C2C というドメイン特性上、最も攻撃価値が高いエンドポイント）。

既存の防御は、(1) Valkey 前段フィルタによる売り切れ後の即時拒否（M-1/M-2）、(2) requestId による冪等性、(3) CloudFront WAF のマネージドルール、(4) 認証系エンドポイントのレート制限（ADR-0012）だが、「売り切れ前の正規在庫を狙う認証済みボットの高频度リクエスト」を直接抑える層がなかった。

レート制限のキー戦略について coordinator が dual-key 方式（user_id 主体 + IP は緩いバックストップ）を提案し、ユーザーが承認済み（Issue #205）。

## 決定

1. **新しいレート制限機構は導入せず、ADR-0012 の `AuthRateLimitService`（Valkey 固定ウィンドウ + Lua 原子化 + fail-open）に endpoint `purchase` を追加して再利用する。**
2. **dual-key 方式**: `secondary` キー = user_id（JWT の sub。認証必須のため常に存在）を**厳しいプライマリゲート**、IP を**緩いバックストップ**とする。
3. 既定閾値（15 分固定ウィンドウ。`AUTH_RATE_LIMIT_PURCHASE_{IP|SECONDARY}` で上書き可能）:
   - **user_id: 10 回 / 15 分** — 正規ユーザーが 15 分に 10 回超の購入試行をすることはない（購入は 1 イベントにつき高々数回。冪等リトライは requestId 再送で別途吸収される）。
   - **IP: 300 回 / 15 分** — 単一 IP からの物量（アカウント使い捨てボット群れ）だけを止める天井。
4. 超過時は 429 + `Retry-After`（既存パターンと同一）。判定は `JwtAuthGuard` 通過後（user_id 確定後）に controller で行う。

## 根拠

- **user_id 主体にする理由**: 発売時のトラフィックは学校・オフィス NAT や大手キャリア CGNAT 経由の相乗りが多く、IP を主キーに厳しい閾値を掛けると正規ユーザーを大量に誤ブロックする。user_id はレート制限の主体として誤爆がなく、認証必須エンドポイントなので常に取得できる。
- **IP を捨てない理由**: user_id だけだと「アカウントを大量に使い捨てるボット群れ」に対して閾値がアカウント数ぶん線形に増える。IP バックストップ（300/15 分）は正規 NAT（数十人が同時購入しても届かない水準）に影響せず、単一 IP からの物量のみを止める。
- **既存 service を再利用する理由**: 固定ウィンドウ + Lua 原子化 + fail-open + 429/Retry-After の挙動・テスト・運用（環境変数上書き）が確立済み。別機構（@nestjs/throttler 等）を足すとレート制限の実装と設定箇所が二重になる。

## 反対材料・トレードオフ

- **固定ウィンドウはウィンドウ境界のバーストを許す**: 境界前後で最大 2 倍の瞬間流量が通る。ADR-0012 と同じ割り切りで、目的（物量の抑制）には十分。
- **fail-open のため Valkey 全断中は制限が消える**: 正確性の正本は PostgreSQL の条件付き更新にあり、レート制限は緩和策という責務分離（ADR-0012 / M-1）を維持する。fail-open の発生は ValkeyFailOpen メトリクス（ADR-0014）で観測できる。
- **IP バックストップは CGNAT の巨大な相乗りでは理論上誤爆し得る**: 300/15 分を超える正規相乗りが観測されたら閾値を環境変数で引き上げる。逆に分散ボットネット（多 IP × 多アカウント）はこの層では止まらず、WAF / CAPTCHA 等の別レイヤの守備範囲とする。
- **user_id 閾値は「購入意図の連打」も数える**: 拒否（sold_out）応答へのリトライ連打も 10 回で止まる。売り切れ後の連打は前段フィルタが既に即時拒否しており、UX 上の影響は小さい。

### 検討した代替案

- **IP 主体の単一キー**: 実装は最少だが、NAT 相乗りの正規ユーザーを誤ブロックする。チケット販売の主要トラフィック源（学校・オフィス・モバイル回線）と正面衝突するため不採用。
- **@nestjs/throttler の導入**: フレームワーク標準だが、Valkey ストレージ・trusted-hops IP 解決・fail-open・Retry-After の挙動を既存実装と二重管理することになる。不採用。
- **イベント単位のグローバル流量制限（inflight cap）**: 人気イベントへの総流量を絞る案。誤爆はないが正規ユーザーも一様に遅くなり、目的（ボット物量の抑制）と合わない。負荷対策としては既存の 3 層（前段フィルタ / Worker 分離 / DB pool 上限）が担う。

## 再検討のトリガー

- 正規ユーザーの 429（user_id 系統）が一定割合で観測されたとき（閾値見直し）。
- 単一 IP 300/15 分を超える正規相乗り（大規模 CGNAT）が観測されたとき（IP 閾値引き上げ、または IP 系統の撤去判断）。
- 分散ボットネットによる購入攻撃が観測されたとき（WAF Bot Control / CAPTCHA 等の上位レイヤ導入検討）。
- 発売瞬間のバースト特性が問題になったとき（固定ウィンドウ → sliding window / token bucket への変更検討）。
