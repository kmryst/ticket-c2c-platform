# 0002. dev 環境を本番系トラックの最初の環境として位置づける

## ステータス

Accepted

## 日付

2026-07-02

## 背景

`docs/poc/technical-validation-plan.md` は、フェーズ 4（クラウド PoC）を「ローカル検証で必要だと分かった場合にのみ実施する」としている。一方で、prod 要件（最大 500 万ユーザー、人気イベントで 100 倍スパイク、影響隔離、在庫超過防止）を満たすシステムへ育てるには、AWS 上の環境を PoC の使い捨てではなく継続的に発展させる場所が必要になる。

「PoC トラックの延長としてクラウド検証環境を作る」のか、「本番系トラック（dev → staging → prod）の最初の環境を作る」のかで、Terraform 設計・CI/CD・運用ルールが大きく変わる。

## 決定

AWS 上に構築する dev 環境は、PoC フェーズ 4 ではなく、**本番系トラック（dev → staging → prod）の最初の環境**として位置づける。

- PoC トラック: Docker Compose によるローカル検証。`docs/poc/technical-validation-plan.md` を正本とする。
- 本番系トラック: AWS 上の dev / staging / prod。`docs/architecture/dev-environment.md` を dev の正本とする。

dev 環境は使わない期間は destroy workflow で削除し、必要なときに Terraform で再構築する運用とする。

## 根拠

- staging / prod へ育てる前提なら、環境分離（state、GitHub Environments、将来の AWS アカウント分離）を最初から設計に織り込む必要があり、使い捨て PoC 構成とは要求が異なる。
- PoC トラックはローカルで完結しており（在庫 PoC 実施済み）、クラウド環境をブロッカーにせず並行して継続できる。
- destroy / 再構築を前提にすることで、常駐コストを抑えつつ「Terraform だけで環境を再現できる」ことを継続的に検証できる。

## 反対材料・トレードオフ

- PoC フェーズ 2（検索）・フェーズ 3（スパイク）のローカル検証より先にクラウド構築へ進むため、Valkey 前段フィルタや SQS FIFO の要否は測定データなしで仮決めになる。ADR-0004 のとおり、可逆な判断（後から追加できる構成）に倒して緩和する。
- 環境を都度 destroy / 再構築するため、環境内に永続させたい検証データは残らない。必要ならシードスクリプトで再投入する。

## 再検討のトリガー

- dev の再構築頻度が高く、destroy 前提運用のほうがコスト高になった場合（常時稼働 + 夜間停止スケジュールへ変更を検討）。
- PoC トラックの測定結果が dev の構成前提を覆した場合（該当 ADR を Superseded にして更新する）。
