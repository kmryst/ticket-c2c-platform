# 0005. API 入口に ALB を採用する

## ステータス

Accepted

## 日付

2026-07-02

## 背景

`docs/architecture/technology-stack.md` の未決事項に「API 入口を API Gateway と ALB のどちらで表現するか」があった。バックエンドは ECS Fargate 上の単一 NestJS サービスで、要件として検索・購入はリスティングの約 10 倍の頻度、人気イベントでは通常の 100 倍のスパイクが想定される。

## 決定

API 入口として ALB を採用する。dev から ALB → ECS Fargate 構成とし、prod では CloudFront + WAF + ALB へ発展させる。API Gateway は採用しない。

## 根拠

1. **ECS の前段には結局 ALB 相当が必要になる。** API Gateway から VPC 内の ECS へは VPC Link 経由で内部 ALB / NLB に到達させる構成が標準であり、API Gateway を選んでも ALB は消えず、レイヤが 1 つ増えるだけになる。
2. **高トラフィック時のコスト構造。** API Gateway HTTP API は約 $1.29/100 万リクエストの従量課金。ALB は時間課金（約 $18/月）+ LCU 課金。平常時 500 RPS（月間約 13 億リクエスト）を仮定すると HTTP API は月 $1,500 超、ALB は LCU 込みで百数十ドル程度と桁が変わり、100 倍スパイクは差をさらに広げる。
3. **スロットル上限。** API Gateway にはリージョン・アカウント単位でデフォルト 10,000 RPS のソフトリミットがあり、100 倍スパイク前提のシステムでは上限緩和の管理が恒常的な運用課題になる。ALB にはこの種のリクエスト上限がない。
4. **API Gateway の付加価値機能を現時点で使わない。** Cognito/JWT オーソライザー（認証は未決・後回し）、API キー・使用量プラン（外部提供なし）、Lambda 統合（バックエンドは ECS）のいずれも該当しない。スロットリングは将来 ALB への WAF レートベースルールで対応する。

## 反対材料・トレードオフ

- **dev 単体のコストだけなら API Gateway が安い。** dev の低トラフィックでは HTTP API は月数セント、ALB は稼働中約 $18/月。それでも ALB にするのは、この dev が prod へ育てる最初の環境であり、prod で選ぶべき入口と同じものを置く（環境パリティ）ため。入口を後から差し替えると、ヘルスチェック、タイムアウト挙動、アクセスログ形式、WAF の付け方がすべて変わる。
- ALB 単体にはレート制限・リクエストバリデーション機能がなく、必要になった時点で WAF（レート制限）とアプリ層（バリデーション）で担う。

## 再検討のトリガー

- 認証を Cognito にし、エッジで JWT 検証したくなった場合。
- 外部開発者向けに API キー・使用量プランが必要になった場合。
- バックエンドを Lambda 中心に寄せる判断をした場合。
