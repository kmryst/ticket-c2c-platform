# 0010. メール+パスワード認証を自前実装（bcrypt + JWT + 自作 Guard）で行う

## ステータス

Accepted

## 日付

2026-07-05

## 背景

購入 API は現状、`purchases.buyer_id` にクライアント申告の UUID をそのまま保存しており、購入者の同一性を何も保証していない（buyer table も存在しない）。requestId バイパス封鎖（M-1、Issue #129）で「入力値は信用しない」方針を徹底した今、残る最大の未検証入力が `buyerId` である。購入フローを本番系に近づけるには、ログイン/認証機能と `users` テーブルの導入が必要になった。

認証の実現方式として、次の 2 案を比較した。

1. **Amazon Cognito（buy）**: マネージド認証。User Pool + Hosted UI / API 連携。
2. **自前実装（build）**: `users` テーブル + bcrypt によるパスワードハッシュ + `@nestjs/jwt` による JWT 発行 + 自作 `JwtAuthGuard`。

また build 案の内部でも、NestJS の事実上の標準である Passport.js（`passport-jwt` / `passport-local`）を使うかどうかを判断する必要があった。

## 決定

メール+パスワード認証を自前実装する。

- パスワードは **bcrypt（コストファクター 12）** でハッシュ化して `users.password_hash` に保存する。
- トークンは **JWT（HS256、有効期限 1h）** とし、素の `@nestjs/jwt` で発行・検証する。リフレッシュトークンは初期スコープに含めない（バックログ）。
- 認可は **自作 `JwtAuthGuard`（`CanActivate` 実装）+ `@CurrentUser()` デコレータ**で行い、Passport.js は使わない。
- 署名シークレットは AWS Secrets Manager + Terraform（`random_password`）で生成し、ECS タスクへ既存の `DB_PASSWORD` と同じ secrets 注入パターンで配備する（Issue #134）。
- バリデーションは class-validator を導入せず、リポジトリ既存流儀の hand-written `unknown` ベース検証で書く。

## 根拠

- **ポートフォリオとしての差別化**: このリポジトリは DevOps / SRE / Platform Engineering のポートフォリオであり、価値の源泉は「認証画面が動くこと」ではなく、build/buy 判断の文書化（本 ADR）、secret の生成・注入・ローテーション設計、監視・検証設計といった周辺運用の成熟度にある。Cognito を貼るとこれらの設計判断がブラックボックスに隠れ、示せるものが減る。
- **既存コードベースとの一貫性**: このリポジトリは TypeORM の query builder を使わず raw `pg`、class-validator を使わず手書きバリデーション、という「仕組みを隠さない手書きスタイル」で統一している。Passport.js の strategy 抽象はこの方針に反し、`CanActivate` を直接実装する方が「Guard で何が起きるか」を 1 ファイルで追える。
- **要件が小さい**: 必要なのは signup / login / me と購入 API の認証必須化のみ。MFA・ソーシャルログイン・パスワードリセットメール等の Cognito が本領を発揮する要件は現時点でない。
- **コストと検証容易性**: 自前実装は追加ランタイムコストがほぼゼロ（Secrets Manager 1 secret ≈ $0.40/月/環境）。単体テストで bcrypt / JWT の全分岐をローカル完結で検証でき、dev 環境のエフェメラル運用（apply → 検証 → destroy）とも相性がよい。
- **bcrypt コスト 12 / HS256 / 1h の選定**: コスト 12 は 2026 年時点の OWASP 推奨帯で、Fargate 0.25 vCPU でも 1 リクエストあたり数百 ms に収まる。鍵共有者が API 1 サービスのみのため非対称鍵（RS256）は不要。有効期限 1h はリフレッシュトークンなし運用での漏洩影響と UX の折衷。

## 反対材料・トレードオフ

- **セキュリティ実装の自己責任**: パスワードポリシー、レート制限、アカウントロック、トークン失効（logout / 漏洩時の即時無効化）はすべて自前になる。特にトークン失効は JWT 単体では原理的にできず、失効リストか短寿命 + リフレッシュトークンの導入が必要。Cognito ならこれらが構成だけで手に入る。
- **機能拡張コスト**: MFA・ソーシャルログイン・パスワードリセットが要件化した場合、自前実装の追加コストは Cognito 比で大きい。
- **コンプライアンス**: 本番でクレデンシャル保管の監査要件（SOC2 等）が課される場合、マネージドサービスに寄せる方が説明コストが低い。
- **Passport 不採用の代償**: NestJS エコシステムの認証サンプル・ライブラリ資産（`@nestjs/passport` 前提のもの）をそのまま流用できない。

## 再検討のトリガー

- MFA・ソーシャルログイン・パスワードリセットメールのいずれかが要件になったとき（Cognito へ移行する損益分岐点）。
- トークンの即時失効（強制ログアウト）が業務要件になったとき（リフレッシュトークン + 失効ストア、または Cognito のトークン失効へ）。
- 本番運用でクレデンシャル保管に外部監査要件が課されたとき。
- 認証系への攻撃（credential stuffing 等）が観測され、レート制限・アカウントロックの自前運用コストがマネージド移行コストを上回ったとき。
