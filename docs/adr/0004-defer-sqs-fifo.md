# 0004. SQS FIFO を初期 dev 環境に含めない

## ステータス

Accepted

## 日付

2026-07-02

## 背景

人気イベントへの購入集中に対して、`MessageGroupId = eventId` による SQS FIFO のイベント単位直列化・流量制御が候補になっている（`docs/architecture/technology-stack.md`）。一方、`docs/poc/technical-validation-plan.md` は「SQS FIFO は最初の PoC には含めない。ローカル検証でイベント単位の直列化や明示的な流量制御が必要だと分かった場合にのみ導入する」と定めており、在庫 PoC の初回結果整理（PR #9）でも「SQS FIFO が必要か」は未判断のまま次の検証候補とされている。

## 決定

初期 dev 環境に SQS FIFO を含めない。購入パスは API → Valkey 前段フィルタ → Aurora 条件付き更新の同期処理とする。

非同期処理（OpenSearch プロジェクション更新など）には SQS Standard + DLQ を使う。Terraform の `sqs` モジュールには `fifo` フラグを持たせ、後から FIFO キューを追加できるようにする。

## 根拠

- FIFO の必要性を示す測定データがまだ存在しない。データなしで導入すると、検証計画（正本）の判断プロセスと矛盾する。
- FIFO を購入パスに入れると購入 API が非同期化し、クライアントへの応答設計（ポーリング / WebSocket / 楽観応答）が必要になる。この複雑さは必要性が確認されてから払うべきコスト。
- 在庫超過防止という正確性要件は Aurora の条件付き更新で既に保証されており（在庫 PoC で oversold = 0 を確認済み）、FIFO は正確性ではなくスループット・隔離のための手段。

## 反対材料・トレードオフ

- 人気イベントのスパイク時、Valkey フィルタを通過したリクエストが Aurora のホット行（同一 eventId の在庫行）に集中し、ロック競合でレイテンシが悪化する可能性がある。dev でのスパイク検証（k6）でこれを測定する。
- 後から FIFO を入れる場合、購入 API の応答設計変更がアプリ側に波及する。導入判断は早めに（スパイク検証の直後に）行う。
- SQS Standard を使う検索 projection では、重複・順序入れ替わりを consumer 側で吸収する必要がある。staging で在庫表示の巻き戻りを 2 回観測したため、version 付き条件更新を [Production Readiness M-10](../architecture/production-readiness.md) で追跡する。これは購入パスの FIFO 導入とは独立した課題である。

## 再検討のトリガー

- dev でのスパイク検証で、人気イベントへの集中負荷が他イベントの購入レイテンシを許容範囲を超えて悪化させた場合。
- Aurora のロック待ち・コネクション枯渇が Valkey フィルタでは抑えられないと測定された場合。

## 測定結果と判定（2026-07-03 追記）

dev 環境への k6 スパイク検証（[スパイク PoC 結果](../poc/spike-poc-results.md)、Issue #76）で上記トリガーを実測した。

観測事実:

- 正確性: 集中負荷下でも oversold は 0（k6 の confirmed 件数と DB 販売数が完全一致）。
- 売り切れ後: 200 req/s × 60 秒の 12,001 リクエストを Valkey 前段が全件拒否し、Aurora への到達は 0 件（p50 10.7ms、エラー 0%）。売り切れ後の保護は完全に機能する。
- 在庫あり期間: 同一イベントへの 200 req/s 集中で、購入スループットが同一総量の分散負荷比で約 1/11 に崩壊し、他イベントの購入も p50 60ms → 5 秒・エラー率 95% に悪化した（トリガー 1・2 の条件に形式上は該当）。原因は同一在庫行の行ロック直列化により伸びたトランザクションが共有 pg pool（max 10）を占有する head-of-line blocking と、Aurora が 20 req/s 時点で既に max 2 ACU に張り付いていた容量不足の複合。

判定: **ステータスは Accepted のまま据え置く（FIFO は引き続き導入しない）**。

- 正確性（oversold 0）と売り切れ後の DB 保護は FIFO なしで成立している。FIFO が解決するのは在庫あり期間のスループット・隔離のみ。
- dev は API 1 タスク（0.25 vCPU）・pool 10・Aurora max 2 ACU の最小構成であり、今回の崩壊は FIFO 以前のサイジング・プール設計の問題が支配的。検証計画（フェーズ 3 メモ）のとおり、この計測は一次判断に位置づける。
- FIFO 導入（購入 API の非同期化）より先に試すべき安価な対策がある: イベント単位の同時実行制限（バルクヘッド）による pool 隔離、pool・タスク数・ACU の適正化、在庫あり期間の前段流量制御。

再検討のトリガー（更新）: staging の `capacity_profile=full`（本番相当サイジング）で上記の安価な対策を入れた上で再測定し、なお人気イベント集中が他イベントの購入 SLO を破る場合、FIFO 導入を新 ADR として起票する。
