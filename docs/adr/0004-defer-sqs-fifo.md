# 0004. SQS FIFO を初期 dev 環境に含めない

## ステータス

Accepted

## 日付

2026-07-02

## 背景

人気イベントへの購入集中に対して、`MessageGroupId = eventId` による SQS FIFO のイベント単位直列化・流量制御が候補になっている（`docs/architecture/technology-stack.md`）。一方、`docs/poc/technical-validation-plan.md` は「SQS FIFO は最初の PoC には含めない。ローカル検証でイベント単位の直列化や明示的な流量制御が必要だと分かった場合にのみ導入する」と定めており、在庫 PoC の初回結果整理（PR #9）でも「SQS FIFO が必要か」は未判断のまま次の検証候補とされている。

## 決定

初期 dev 環境に SQS FIFO を含めない。購入パスは API → Valkey 前段フィルタ → Aurora 条件付き更新の同期処理とする。

非同期処理（OpenSearch プロジェクション更新など）には SQS Standard + DLQ を使う。Terraform の `sqs` モジュールには `fifo` フラグを持たせ、後から FIFO キューを追加できるようにする。

## 根拠

- FIFO の必要性を示す測定データがまだ存在しない。データなしで導入すると、検証計画（正本）の判断プロセスと矛盾する。
- FIFO を購入パスに入れると購入 API が非同期化し、クライアントへの応答設計（ポーリング / WebSocket / 楽観応答）が必要になる。この複雑さは必要性が確認されてから払うべきコスト。
- 在庫超過防止という正確性要件は Aurora の条件付き更新で既に保証されており（在庫 PoC で oversold = 0 を確認済み）、FIFO は正確性ではなくスループット・隔離のための手段。

## 反対材料・トレードオフ

- 人気イベントのスパイク時、Valkey フィルタを通過したリクエストが Aurora のホット行（同一 eventId の在庫行）に集中し、ロック競合でレイテンシが悪化する可能性がある。dev でのスパイク検証（k6）でこれを測定する。
- 後から FIFO を入れる場合、購入 API の応答設計変更がアプリ側に波及する。導入判断は早めに（スパイク検証の直後に）行う。

## 再検討のトリガー

- dev でのスパイク検証で、人気イベントへの集中負荷が他イベントの購入レイテンシを許容範囲を超えて悪化させた場合。
- Aurora のロック待ち・コネクション枯渇が Valkey フィルタでは抑えられないと測定された場合。
