# 0001. ADR で設計判断を記録する

## ステータス

Accepted

## 日付

2026-07-02

## 背景

CLAUDE.md は「トレードオフを伴う設計判断は、後で ADR として残す候補にする。現時点では ADR ディレクトリは未作成のため、必要になった時点で `docs/adr/` と ADR 運用ルールを追加する」と定めていた。

prod 要件を前提にしたクラウドバックエンド dev 環境の構築に着手するにあたり、API 入口の選定、Terraform state 設計、Worker 実行形態など、変更コストが高くトレードオフを伴う判断が複数発生する。この時点が「必要になった時点」に該当する。

## 決定

`docs/adr/` を作成し、[README.md](./README.md) のフォーマットと運用ルールに従って設計判断を記録する。

## 根拠

- 判断の背景と反対材料を残すことで、後から「なぜこうしたのか」を再現できる。
- 再検討のトリガーを明文化することで、状況が変わったときに判断を見直す起点になる。
- `docs/architecture/` の正本ドキュメントは「現在の姿」を書く場所であり、「判断の経緯」を書く場所を分けたほうが両方が読みやすい。

## 反対材料・トレードオフ

- ドキュメントの置き場所が増え、正本ドキュメントとの二重管理リスクが生じる。README に「ADR で判断が変わったら正本も同じ PR で更新する」ルールを置いて緩和する。

## 再検討のトリガー

- ADR の件数が増えて一覧管理が破綻した場合（カテゴリ分け・ツール導入を検討）。
