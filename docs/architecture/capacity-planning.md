# 負荷シナリオと容量計画

## ステータス

ドラフト・未着手。[production-readiness.md](production-readiness.md) の L-21（負荷シナリオを定義したドキュメントがなく、Issue #113（L-8 の Postgres 接続プール対応）が想定トラフィック未確定を理由に未着手のまま残っている）の対応として、Issue #289 で新設した。

このドキュメントはシナリオごとの「実測データ」と「未検証の想定」を区別して記録するものであり、値を確定させるものではない。ここに書かれた「要検証」項目を実施し、実測データが埋まった段階で初めて Issue #113（L-8）の対応方針判断に進める。

## 前提の性質に関する注記

本ドキュメントで参照する [technology-stack.md](technology-stack.md) 記載の数字（最大 500 万人ユーザー、人気イベントは通常の約 100 倍のトラフィック、検索・購入はイベントリスティングの約 10 倍の頻度）は、いずれもシステム設計課題における設計時の想定であり、出典（ユーザーヒアリング記録、実測ログ、外部ベンチマーク等）の記載はない。

DevOps / SRE / Platform Engineering の容量計画としては、これらの数字を無条件の確定値として扱わず、実測データがある項目と、設計上の想定に留まる項目を区別したうえで、後者には「値を確定させるために何を検証する必要があるか」を明記する。本ドキュメントの表・prose では、値の性質を次のタグで区別する。

- **実測**: k6 等による実際の負荷試験で得られた数値。試験条件・出典（Issue 番号、実施日）を併記する。
- **想定（未検証）**: `technology-stack.md` 等に記載された設計時の想定値、またはそれらから導出した値。出典と、値を確定させるために必要な検証手段を併記する。
- **算出**: 必要構成（ECS タスク数・Aurora ACU 等）が既に確定している項目について、AWS 公式単価（AWS Price List API、`ap-northeast-1`、2026-07-13 時点）× 既知の構成値から計算した概算コスト。実際の請求実績（実測）ではないが、単価・計算式の出典は明記する。トラフィックに応じてタスク数・ACU が変動する範囲がある場合は下限〜上限のレンジで示す。
- **未算出**: 必要構成自体が未確定（想定のみ）のため、コストを計算する前提がそろっていない項目。構成確定後に算出する旨を明記する。

### コスト計算の単価出典

AWS Price List API（`aws pricing get-products`、2026-07-13 取得、`ap-northeast-1`）で確認した単価は次のとおり。

| 項目 | 単価 |
| --- | --- |
| Fargate vCPU（Linux/x86_64） | $0.05056 / vCPU 時間 |
| Fargate メモリ | $0.00553 / GB 時間 |
| Aurora Serverless v2（PostgreSQL 互換） | $0.15 / ACU 時間 |

API/Worker/Frontend の ECS タスクは `terraform/modules/ecs-service` の既定値（`cpu=256`=0.25 vCPU、`memory=512`=0.5 GB）を使用しており、staging はこの既定値を override していない。1 タスクあたりの時間単価は次のとおり。

`0.25 vCPU × $0.05056 + 0.5 GB × $0.00553 = $0.01264 + $0.002765 ≒ $0.0154 / タスク時間`

staging full（ADR-0018）の desired_count / autoscaling min-max は次のとおりで、これがシナリオ①〜③に共通する ECS タスク数のレンジになる。

- 定常時（min）: API 2 + Worker 2 + Frontend 2（固定）= 6 タスク → `6 × $0.0154 ≒ $0.092 / 時間`
- autoscaling 上限まで到達した場合（max）: API 4 + Worker 4 + Frontend 2（固定）= 10 タスク → `10 × $0.0154 ≒ $0.154 / 時間`

Aurora full（ADR-0018 / staging-environment.md）は min ACU 0.5 / max ACU 8 のため、ACU 側のコストレンジは次のとおり。

- 下限（min ACU 0.5）: `0.5 × $0.15 = $0.075 / 時間`
- 上限（max ACU 8）: `8 × $0.15 = $1.20 / 時間`

このため、①〜③の「ECS + Aurora」概算コストは理論上 **$0.167/時間（定常・下限）〜 $1.354/時間（autoscaling 上限・Aurora 上限まで到達した場合）** のレンジになる。実際にこのレンジのどこに収まるかは、CPU 使用率・ACU 消費量の実測（本ドキュメントの「要検証」で挙げている staging full 再試験）で確定する。Valkey / OpenSearch / NAT Gateway / ALB など、4 シナリオで共通して稼働する基盤コンポーネントのコストは、シナリオ間で差分がないためこの概算には含めない（`dev-environment.md`「コスト目安」と同様の方法で見積る場合は別途参照）。

### ① baseline（平常時混合負荷）

- **実測**: staging full（2026-07-04、Issue #93）で `MODE=baseline BG_RATE=20 DURATION=60s` を実行し、p50=41.9ms / p95=68.7ms / p99=178.4ms、エラー率 0% を確認済み。これは k6 試験条件としての 20rps であり、実運用の想定 RPS ではない。
- **想定（未検証）**: `technology-stack.md`（29〜33 行目）は「最大 500 万人のユーザーを想定する」「検索と購入はイベントリスティングの約 10 倍の頻度」とするが、これらから実運用の同時接続数・RPS を逆算した値は存在しない。DAU 率・ピーク集中率などの前提が未定義のため、20rps という試験条件が平常時の実態に対して大きいのか小さいのかも判断できない。
- **算出（コスト）**: 定常構成（API 2 + Worker 2 + Frontend 2 = 6 タスク、Aurora 0.5 ACU）で $0.167/時間。baseline は実測でエラー率 0%・レイテンシも低いため autoscaling が発火する負荷ではなく、定常構成のまま推移する可能性が高いが、CPU 使用率・ACU 消費量そのものは今回測定していない。
- **要検証**:
  1. プロダクト側へのヒアリングにより、登録ユーザー数の実績見込み、DAU 率、1 日あたりの平均アクセス回数、ピーク時間帯への集中率を確認する。
  2. 上記から `想定ピーク RPS ≒ DAU × 1 ユーザーあたり平均リクエスト数 ÷ ピーク集中秒数` の形で逆算し、baseline の想定トラフィックとして記録する。
  3. 逆算した RPS で改めて staging full の baseline 試験を実施し、実測値を更新する（CPU 使用率・ACU 消費量も観測項目に加え、コスト算出のレンジを狭める）。

### ② 人気イベント発売直後 spike

- **実測**: staging full（2026-07-04、Issue #93）で `HOT_RATE=200 BG_RATE=20 DURATION=60s` を実行し、p50=4.6〜4.7s / p95=7.4〜7.5s / p99=7.6〜7.9s、エラー率約 29〜30% を確認済み。原因は API 側 `pg.Pool({ max: 10 })`（1 タスクあたり）× 2 タスク稼働 = 合計 20 接続の枯渇によるタイムアウトであり、アプリのバグではないと特定済み（`production-readiness.md` L-8）。oversold（在庫超過）は 0 件を維持。
  - この試験は ECS autoscaling policy（ADR-0018、2026-07-10 導入）より前に実施されたものであり、CPU 使用率・オートスケール発火有無は観測していない（`production-readiness.md` L-21 本文）。`connectionTimeoutMillis: 5000` によりプール枯渇時のリクエストは CPU 負荷を上げずに 5 秒でタイムアウトする構造のため、CPU 60% 閾値の target-tracking policy が今回のボトルネックに対して発火しない可能性がある。
- **想定（未検証）**: `technology-stack.md`（30 行目）の「人気イベントでは通常の約 100 倍のトラフィックが発生する可能性がある」は出典不明の想定値。①の baseline RPS が未確定のため、「100 倍」に相当する RPS も算出できず、実測した 200rps がこの想定倍率に対して過小か過大かは不明。
- **算出（コスト）**: 定常構成（$0.167/時間）〜 autoscaling 上限まで到達した場合（API/Worker 各 4 タスク + Aurora 8 ACU、$1.354/時間）のレンジ。この試験は ADR-0018 導入前に実施されたため、実際に CPU 使用率が 60% 閾値へ到達し autoscaling が発火したか、Aurora ACU がどこまで消費されたかは観測しておらず、レンジのどこに実態が収まるかは未確定（L-21 本文が指摘する「プール枯渇時のタイムアウトは CPU 負荷を上げずに失敗する構造」のとおりであれば、autoscaling が発火せず定常構成側（$0.167/時間）に近いまま同じエラー率悪化が再現するリスクがある）。
- **要検証**:
  1. ①の baseline RPS が確定した後、「約 100 倍」に相当する RPS を算出し直す。
  2. staging full（ADR-0018 の autoscaling 有効化後の構成）で同条件の k6 spike を再実行し、CPU 使用率と ECS タスク数推移（autoscaling 発火有無）、Aurora ACU 消費量を新規の観測項目として追加する（`production-readiness.md` L-21 対応コスト (b)）。プール枯渇によるタイムアウトが CPU 使用率を実際に押し上げるのか、押し上げないまま同じ現象が再現するのかを確認するとともに、上記コストレンジのどこに実態が収まるかを確定させる。

### ③ 売り切れ後継続リクエスト

- **実測**: staging full（2026-07-04、Issue #93）で `HOT_RATE=50 BG_RATE=10 DURATION=30s`（hot 残数 330 消化条件）を実行し、p50=10〜41ms / p95=179〜249ms / p99=647〜785ms、エラー率 0% を確認済み。Valkey 前段拒否（`purchase_rejected_precheck`）1171 件に対し、Aurora への到達（`purchase_rejected_db`）は 0 件で、売り切れ後の影響隔離が正常に機能していることを確認済み。
- **想定（未検証）**: この試験は k6 の機械的なリクエスト送出パターンであり、実際のプロダクト運用開始後にユーザーがどの程度の頻度で完売イベントへ再アクセスするか（手動リロード、bot・スクレイパーの有無）は想定のみで実測ログの裏付けがない。50rps という条件自体も想定トラフィックではなく試験設計上の値。
- **算出（コスト）**: 定常構成で $0.167/時間。Valkey 前段拒否により Aurora 到達が 0 件であることを実測で確認済みのため、Aurora 側の追加消費（ACU 上昇）は想定しにくく、②と異なりレンジの下限に近い値で推移する可能性が高い。ただし高レート条件（要検証 1.）でも同様に下限で収まるかは未確認。
- **要検証**:
  1. より高いレート（200rps 以上、②の spike 条件を売り切れ後まで継続するパターン）で、Valkey 前段拒否が高負荷下でも定常的に機能し続けることを確認する。あわせて CPU 使用率・ACU 消費量を観測し、上記コスト想定（下限で推移）を裏付ける。
  2. 実運用開始後、実際のアクセスログから完売イベントへのアクセスパターン（頻度・継続時間・bot 比率）を分析し、本シナリオの前提トラフィックを実測値に置き換える。

### ④ 複数人気イベント同時発売

- **実測**: なし。既存の k6 スクリプト（`scripts/load-testing/purchase-spike.js`）は `HOT_EVENT_ID` が単一イベントの UUID を前提とした実装であり、hot イベント複数への同時集中負荷には対応していない。
- **想定（未検証）**: 複数の人気イベントが同時期（同日・同時刻）に発売される運用があるかどうか自体が `technology-stack.md` に記載がなく、プロダクト側の運用想定も本ドキュメント作成時点では確認していない。「人気イベント集中が他イベントの性能を劣化させない必要がある」という要件（technology-stack.md 31 行目）は hot が単一の場合のみ検証済みで、hot が複数の場合（Postgres 接続プール消費・Valkey 前段フィルタ処理・ホット行競合が加算される）は未検証。
- **未算出（コスト）**: 必要構成自体が「②と同一構成を仮定」という未検証の前提のため、コストを計算する土台がない。hot が複数になった場合に API/Worker の autoscaling max（現状 4）で足りるのか、Aurora ACU 上限（現状 8）で足りるのかは②とは別に検証が必要であり、構成が変われば単価計算（本ドキュメントの「コスト計算の単価出典」と同じ方法）もやり直す。
- **要検証**:
  1. プロダクト側へ、同一時間帯に複数の人気イベントが発売される運用の有無・頻度をヒアリングする。
  2. 同時発売がありうる場合、k6 スクリプトを複数 `HOT_EVENT_ID` に対応させる改修を行う。
  3. staging full で hot 2〜3 イベント + background 同時実行の試験を実施し、単一 hot（②）との比較でボトルネックが加算的に悪化するかを確認する。

## 今回のスコープ外

以下の 3 区分は、負荷シナリオの検討過程で候補に挙がったが、Issue #113 / L-21 の直接のブロッカー解消には必須でないと判断し、今回の `capacity-planning.md` の対象外とした。必要になった時点で別 Issue として起票する。

- **認証バースト（発売直前の一斉ログイン・リフレッシュ集中）**: bcrypt によるパスワード検証は CPU 集約的で、Postgres 接続プール枯渇とは異なる経路で CPU ベースの autoscaling（ADR-0018）を発火させうる、購入フローとは独立した懸念のため、別スコープとして扱う方が検証設計をシンプルにできる。
- **読み取り系バースト（検索・イベント一覧・SSR ページ閲覧の集中、購入を伴わない）**: `technology-stack.md` の「検索と購入はイベントリスティングの約 10 倍の頻度」という前提自体が実測の裏付けのない想定値であり、この前提を検証しないまま読み取り系の負荷シナリオを表に含めると、根拠のない数字の上にさらに数字を積み上げることになる。まず購入系コア 4 区分と baseline の想定 RPS 確定を優先し、読み取り系は前提の検証状況を見て別途検討する。
- **soak（持続負荷、1 時間以上の継続試験）**: 接続リーク・Aurora ACU の追従・Worker projection ラグの蓄積などスパイク試験では見えない劣化を検出する目的だが、staging full を長時間稼働させるコスト判断（ephemeral destroy 運用の前提との兼ね合い）が別途必要であり、コア 4 区分と切り離して検討する方が適切。

## L-8 / L-21 との対応関係

`production-readiness.md` の L-8（Postgres 接続プール上限による spike 時のエラー率悪化）は、対応の選択肢（プールサイズ増、Aurora ACU 上限引き上げ、RDS Proxy 導入）を「想定同時接続数（実トラフィック見込み）を確定させる」ことを前提に保留してきた（Issue #113）。

本ドキュメントは、この「想定トラフィック未確定」という保留理由を、シナリオ区分ごとに「何が実測済みで、何が未検証の想定に依存しているか」に分解して可視化するものである。特に②（人気イベント発売直後 spike）は L-8 の実測データそのものを引き継いでいるが、その実測は ADR-0018 の autoscaling 導入前に行われたものであり、CPU 使用率・オートスケール発火有無という新しい観測項目を追加した再試験（本ドキュメントの「要検証」参照）が済むまでは、L-8 の対応方針（プールサイズ増か、ACU 引き上げか、RDS Proxy 導入か）を判断する材料が揃わない。

①（baseline）の想定 RPS が確定すれば、②の「約 100 倍」という想定倍率も実際の RPS に換算でき、L-8 の対応コストとリスクを具体的な数字で比較できるようになる。この 2 点（baseline RPS の確定、staging full での CPU/autoscaling 観測を含む再試験）が揃うことが、Issue #113 の着手判断の前提条件である。
