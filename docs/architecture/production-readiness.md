# Production Readiness バックログ

## ステータス

dev / staging の監査・実地検証と B2C 目標設計のレビューで見つかった、未対応の Production Readiness 課題を記録する。

ここに記載する項目は、dev / staging で意図的に許容している暫定措置、B2C 実装時に設計・検証する事項、または prod 展開前に対応が必要な既知のギャップである。トレードオフを伴う設計判断そのものは `docs/adr/` に ADR として記録し、このドキュメントは「まだ決めていない・直していない」項目のバックログとして運用する。

対応が完了した項目は削除せず、ステータス欄を更新する。対応済み項目の詳細な実装・実地検証ログ（検証手順・実測値・教訓・追記履歴）は [production-readiness-log.md](production-readiness-log.md) に移設しており、本ドキュメントのテーブルには要約と該当ログへのリンクのみを残す。

## 次の優先順位（推奨）

上記のうちどれから着手すべきかの推奨順位。2026-07-19、L-19 の dev / staging 実地確認と destroy 後確認の完了を反映して更新。旧優先項目（H-1、M-1 / M-2、L-5、L-15）と dev / staging 向け Observability の仕上げは対応済みであり、残作業は M-9 の rollout 検証・CI/CD hardening・負荷設計・prod 常設運用向けの見直しに分かれる。

1. **M-14**: WAF ログから Cookie / Authorization を redaction する。トークン値をログへ複製する理由がなく、B2C のトラフィック・SLO の確定を待たない。
2. **M-13 / M-16 / L-1 / L-2**: ECR artifact の不変性と scan gate、deploy / migration の最小権限 role、GitHub Actions の SHA pin、plan role の subject を順に harden する。CI/CD の入口が侵害された際の artifact 改竄と infrastructure blast radius を下げる系列として扱う。
3. **M-15**: Aurora の migration 管理権限と API runtime 権限を分け、公開 API から DB 管理権限へ到達しないようにする。
4. **M-10 → M-12**: version 付きの条件更新で OpenSearch projection の巻き戻りを防いだ後、安全な replay を前提に event publish / target delivery の欠損検知・保全・復旧方式を整備する。staging で 2 回再現した整合性問題と、commit / publish 間に構造上残る欠損リスクを同じ projection reliability の作業系列として扱う。
5. **M-19 / M-9**: migration 互換性、post-deploy smoke、rollback 判断を整えたうえで、Interface Endpoint 3 種撤去を dev / staging へ rollout し、backend / frontend deploy、ECS の安定稼働、CloudWatch Logs の到達を確認する。
6. **M-17 / M-18 / M-20**: B2C のトラフィック、RPO / RTO、保存要件、account-level platform baseline の責任範囲を確定し、prod 基盤と restore rehearsal を設計する。
7. **L-21 → L-8**: 想定同時アクセス数・RPS と staging full の再負荷試験結果を確定してから、Postgres 接続プールサイズ・Aurora ACU 上限・RDS Proxy の選択を判断する。
8. **L-23 → L-26 / L-25 / L-34**: OpenSearch IAM、service SG、Valkey RBAC、app egress、Flow Logs を組み合わせて、task identity と east-west / outbound 通信の最小権限・可観測性を整える。
9. **L-27 / L-28 / L-30 / L-33**: B2C frontend の CSP、bot / rate 制御、Worker 停止検知、常設運用のコスト検知は、実トラフィックと購入 SLO が確定してから方式と閾値を設計する。
10. **L-6 / L-20 / L-29 / L-31 / L-32 / L-22**: データ層 SG egress、prod SRE 運用、scheduled cleanup 監視、Dependabot coverage、drift / policy-as-code を整える。L-22 の `Frontend Build` / `Playwright E2E` required 化は意図的に保留し、現時点では変更しない。

**期限管理**: M-11 は上記の相対的な優先順位とは別に、2027-06-04 を完了期限とする。ほかの作業状況を理由にこの期限を後ろ倒ししない。
