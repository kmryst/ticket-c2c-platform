# 技術検証計画

## ステータス

ドラフト。

このドキュメントは、C2C チケット販売プラットフォーム設計における高リスク領域を検証するための初期 PoC 計画を定義するものです。プロダクト全体の実装計画ではありません。

## ドキュメント名

DevOps、SRE、Platform Engineering の文脈では、この種の文書には次のような名前が使われる。

- Technical Validation Plan
- PoC Plan
- Spike Plan
- Experiment Plan
- Performance Test Plan
- Architecture Validation Plan

このプロジェクトでは **Technical Validation Plan** を採用する。目的はプロダクトのプロトタイプを作ることではなく、アーキテクチャ上のリスクを測定可能な結果で検証することだからです。

ADR は、検証結果から設計判断を行った後に使う。

## 背景

対象システムは C2C チケット販売プラットフォームであり、次の重要な制約を持つ。

- 最大 500 万人のユーザーを想定する。
- 人気イベントでは通常の約 100 倍のトラフィックが発生する可能性がある。
- 人気イベントへのトラフィック集中が他イベントを劣化させてはならない。
- チケット予約は在庫数を超えてはならない。
- 検索と購入は、イベントリスティングの約 10 倍の頻度で行われる想定。

## 検証目的

PoC では、システム設計に最も影響する次のリスクを検証する。

1. 同時購入リクエスト下での在庫正確性。
2. Valkey を在庫前段フィルタとして使った場合のデータベース負荷低減。
3. 位置情報、イベント種別、イベント開催日、将来追加される検索条件に対する検索実現性。
4. 1 つの人気イベントにトラフィックが集中した場合のスパイク挙動。
5. 1 つの人気イベントが無関係なイベントを劣化させないための影響隔離。

## 非目的

初期 PoC では、本番システム全体を構築しようとしない。

初期 PoC のスコープ外:

- 本番 UI。
- 認証・認可。
- 決済処理。
- Seller 管理 UI。
- 購入キャンセル。
- CloudFront、WAF、Route 53、API Gateway。
- ECS Fargate への本格的な自動デプロイ。
- Terraform による本格的な本番インフラ。
- 完全な可観測性基盤。

## 境界

本番アーキテクチャでは、アプリケーション入口として ALB を使う想定がある。

ただし、最初の PoC は Docker Compose を使ったローカル検証から始める。中核リスクは AWS リソースを構築しなくても検証できるためです。

初期ローカル検証の境界:

- API サービス。
- PostgreSQL。
- Valkey。
- k6 負荷テスト。

必要になった場合のクラウド検証境界:

- ALB。
- ECS Fargate。
- Aurora PostgreSQL。
- ElastiCache Valkey。
- OpenSearch。
- SQS。
- EventBridge。
- CloudWatch logs。

## 推奨実施順序

### フェーズ 1: 在庫 PoC

目的:

同時購入リクエストによって過剰予約が発生しないことを検証する。

初期コンポーネント:

- NestJS API または最小構成の TypeScript サービス。
- PostgreSQL。
- Valkey。
- k6。

シナリオ:

- Valkey を使わず、PostgreSQL の条件付き更新だけで購入する。
- Valkey 前段フィルタと PostgreSQL 最終確定を組み合わせて購入する。
- 売り切れ後のイベントに大量の追加購入リクエストを送る。
- 複数イベントのうち、1 つの人気イベントにトラフィックを集中させる。

メトリクス:

- 成功した購入数。
- 拒否された購入数。
- 最終的な残在庫。
- 過剰予約件数。
- PostgreSQL 更新試行数。
- PostgreSQL のロック待ち、またはクエリレイテンシ。
- API レイテンシ p50、p95、p99。
- エラー率。

受け入れ基準:

- 確定購入数が設定在庫数を超えない。
- 過剰予約件数が常に 0 である。
- Valkey により、売り切れ後の不要な PostgreSQL 到達数が減る。
- 同一テスト内で、人気イベント負荷が無関係なイベントのレイテンシを大きく悪化させない。

### フェーズ 2: 検索 PoC

目的:

イベント検索が、位置情報、イベント種別、イベント開催日、将来追加される検索条件に対応できることを検証する。

初期コンポーネント:

- PostgreSQL を正本とする。
- OpenSearch を検索用プロジェクションとする。
- 簡易同期ワーカーまたはスクリプト。

シナリオ:

- 位置情報で検索する。
- イベント種別で検索する。
- イベント開催日で検索する。
- 位置情報、イベント種別、イベント開催日を組み合わせて検索する。
- 将来追加される想定のフィルタを 1 つ加え、インデックス拡張性を確認する。

メトリクス:

- インデックス更新レイテンシ。
- 検索レイテンシ p50、p95、p99。
- 検索結果の正確性。
- PostgreSQL から OpenSearch へのプロジェクション遅延。

受け入れ基準:

- 必須フィルタに対して、検索が正しい一致イベントを返す。
- クエリパターンごとに主データベーススキーマを変更しなくても、将来フィルタを追加できる見通しがある。
- プロジェクション遅延を測定でき、イベント一覧・検索用途として許容できる。

### フェーズ 3: スパイク PoC

目的:

1 つの人気イベントに大きなトラフィックスパイクが発生した場合の挙動を検証する。

シナリオ:

- 1 つのイベントに対して 100 倍相当の購入試行を行う。
- 同時に他イベントへ通常量の購入試行を行う。
- 購入スパイク中に検索トラフィックを流す。
- 在庫枯渇後に売り切れリクエストを流す。

メトリクス:

- 人気イベントの購入レイテンシ。
- 他イベントの購入レイテンシ。
- 購入スパイク中の検索レイテンシ。
- PostgreSQL コネクション使用量。
- PostgreSQL クエリレイテンシ。
- Valkey コマンドレイテンシ。
- API エラー率。

受け入れ基準:

- 人気イベントのスパイク中も、他イベントが利用可能である。
- 売り切れリクエストが速やかに拒否される。
- PostgreSQL が回避可能な売り切れトラフィックから保護される。
- SQS FIFO または別の流量制御の仕組みが必要かどうかを、結果から判断できる。

### フェーズ 4: 必要な場合のクラウド PoC

目的:

ローカル検証で有用な結果が出た後に、AWS 固有の実行時挙動を検証する。

候補コンポーネント:

- ALB。
- ECS Fargate。
- Aurora PostgreSQL。
- ElastiCache Valkey。
- OpenSearch。
- SQS FIFO または Standard。
- EventBridge。
- CloudWatch logs。

このフェーズは、ローカル結果からインフラ挙動、ネットワークレイテンシ、マネージドサービスの制限、運用時挙動の測定が必要だと分かった場合にのみ実施する。

## 初期アーキテクチャ仮説

初期の在庫設計では、PostgreSQL を最終的な正本とする。

Valkey は高速な前段フィルタとして使う。

1. 購入リクエストを受け取る。
2. リクエストを検証する。
3. Valkey の在庫カウンタを確認・減算する。
4. 売り切れの場合は即時拒否する。
5. PostgreSQL の条件付き更新で購入を確定する。
6. PostgreSQL での確定に失敗した場合は、Valkey カウンタを補償する。
7. 測定用に結果を記録する。

SQS FIFO は最初の PoC には含めない。ローカル検証でイベント単位の直列化や明示的な流量制御が必要だと分かった場合にのみ導入する。

## 判断材料として出すもの

PoC では、後続の設計判断を支えるために次のデータを出す。

- 想定する購入パスに対して PostgreSQL の条件付き更新で十分か。
- 購入パスで Valkey を必須にすべきか。
- 人気イベントに SQS FIFO が必要か。
- 検索で最初から OpenSearch を使うべきか。
- アーキテクチャ確定前にクラウド検証が必要か。

## 作成する成果物

PoC 作業から期待する成果物:

- 負荷テストスクリプト。
- テストデータ生成メモ。
- p50、p95、p99 レイテンシとエラー率を含む結果サマリー。
- 在庫正確性レポート。
- 検索正確性レポート。
- 測定結果に基づく判断メモまたは ADR。
